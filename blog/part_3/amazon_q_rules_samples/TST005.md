# TST005: Test Method Naming Convention
## Purpose
This rule defines a clear naming convention for test methods to ensure they are self-documenting and immediately convey what is being tested, what the expected outcome is, and under what conditions.

## Instructions
- Use a three-part naming convention for test methods:
  1. **Method being tested** - Clearly identify which method/function is under test
  2. **Expected outcome** - Describe what should happen (returns, raises, sets, etc.)
  3. **Significant input condition** - Describe the key input or state that triggers this behavior

- Format: `test_{method}_{outcome}_when_{condition}`
- Use descriptive verbs for outcomes: `sets`, `returns`, `exits`, `raises`, `creates`, `succeeds`, `selects`, etc.
- Use clear condition descriptions that explain the significant input or state
- For simple initialization tests, condition may be omitted: `test_init_sets_correct_path`
- Be specific about conditions to distinguish similar scenarios

## Priority
HIGH

## Valid Examples
```python
# Good - follows all three rules
def test_init_sets_correct_path(self) -> None:
    # Tests: init method, sets path, no special condition needed

def test_invoke_task_exits_when_no_templates_in_config(self) -> None:
    # Tests: invoke_task method, exits with error, when config has no templates

def test_invoke_task_exits_when_template_not_found_in_config(self) -> None:
    # Tests: invoke_task method, exits with error, when requested template not in config

def test_invoke_task_succeeds_when_changeset_created(self) -> None:
    # Tests: invoke_task method, succeeds, when changeset is successfully created

def test_invoke_task_selects_correct_template_when_multiple_templates_configured(self) -> None:
    # Tests: invoke_task method, selects correct template, when multiple templates available

def test_invoke_task_creates_stack_manager_with_execution_role_when_collection_has_execution_role_and_template_does_not(self) -> None:
    # Tests: invoke_task method, creates StackManager with execution role, when collection has role but template doesn't
```

## Invalid Examples
```python
# Bad - only describes input, not outcome
def test_invoke_task_template_not_found(self) -> None:

# Bad - vague outcome description
def test_invoke_task_error(self) -> None:

# Bad - missing condition when it's significant
def test_invoke_task_exits(self) -> None:

# Bad - too verbose
def test_invoke_task_method_should_exit_with_error_code_when_template_is_not_found_in_configuration(self) -> None:
```

## Benefits
- Test names immediately convey the complete test scenario
- Easy to understand test failures from test runner output
- Self-documenting test suite
- Consistent naming across the codebase
- Easier to identify missing test coverage
- Distinguishes between similar scenarios (e.g., different execution role inheritance patterns)
